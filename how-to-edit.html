How to edit (for dummies)(you)

  • A guide, be it yours or one to import
  • A browser
  • A Desktop PC/Laptop (sorry phone people)

    • Heavily recommended:
  • Visual Studio Code (VS Code)
    • Opening HTML files in it will highlight code, making it much more readable.

    this is what this very website looks in it!

    Step 1

    Define what your guide is exactly: is it a meta set compilation for rise, or a tutorial on how to get into speedrunning?
    Once you've done that, find the folder your guide will be in; it should be self-explanatory, but if you don't understand it, ask and I'll add a section on to how to do that.

    Step 2

    Add all your planned pictures in .webp format to the "pictures" folder (might be labelled differently), and make sure their names are in the same format as the other files in there. The first image in the Switch Axe document is "switch_axe1", the third image in the Heavy Bowgun document is "heavy_bowgun3", and so on. Make sure you don't capitalize: everything is case-sensitive.

    Step 3

    Most of your work from here will be Copy-Pasting. The general Structure of Weapon-related guides is done, you just need to paste the words in the labelled spots and make sure the Images you want to use have the EXACT NAME as found within the html.
    If the image reference says "swaxe1" for example, name it exactly that. Don't get fancy with capitalization, as this whole thing is extremely case sensitive. Don't add spaces either.
    Example of how that's supposed to look:


    Step 4

    If you want to check your work, open a new tab (or better: window) in your browser, then press CTRL+O to open a file and select the html file you wish to check, the same way you opened this locally. Here, you can edit as much as you'd like as it's purely visible to you. Make sure you grab screenshots of the tables you use, as they tend to break on mobile view (which the majority of people will use).

    <span class="ClassNameHere"></span> <-- these things are called tags. Most tags you open will need to be closed again, too, otherwise the site might break. VS code is useful here, as you can minimize the tags you used which shows you what all is considered as part of that tag, making troubleshooting easier.
    Generally speaking, as long as you copy-paste properly and only fill in the obviously labelled spots, it'll work fine.
    If you need to point at a stylesheet or a script, we use something called relative links. They work by telling the html code the name of the file you're trying to access. If that file is in a different folder, you'll need to tell it to look in that folder, like so:
    <link href="Folder/File.type">
    If you need to go "up" a folder, you indicate this by using ../, like: href="../Folder/Folder/File.type", still in the funni brackets.
    Your main use case for this will be when embedding images, like so:
    <img src="Sitewide_Images/vssample.GIF" class="BuildCard">, which will display it as such:

    Note that since we host on GitHub, EVERYTHING IS CASE SENSITIVE. This will be your second most common error.


    Make sure, absolutely sure, that:

  • Files have the correct name

  • Files are in the correct folder

  • Files have the correct format

  • If you work with how to play guides: your videos on youtube are about 3 minutes long, unlisted and in the correct playlist.

    • You navigate to the location your guide belongs to, then open the pre-prepared html file there. Copy the asset you need, paste it where you need it and fill out the obviously marked spots. That's it!

    As previously mentioned, the second most common error you will do is capitalization of filenames that are not capitalized.
    The most common thing you will do is either forgetting to remove comments (shown green in vs code), or not closing a div.
    You test for those by navigating to the folder the file you refer to, most commonly: images are located at and check their spelling versus yours.
    For forgotten closed divs or spans, you can click on the tag (they're blue) and it'll show you where its closed, or on the little arrow next to it to minimize everything it encapsulates.
    If it goes on for far longer than you want it to, slap your closing tag after where you want it to close.
    Another common mistake you will make is referring to folders that aren't there, by forgetting to go up enough (or going up too many) steps with ../ - this too you can fix by just checking where everything actually is.
    Note that this type of link is called relative link, as it's relative to the file you're typing in, in your case: 99 times out of 100, the html file, so you need to follow the path from there.
    The last easy to fix but important to remember mistake is re-using ids. An id is a one-time-use name, given to a specific object. You will find them in the Collapsible Headers, where you'll need to make sure that the input id="THIS" and the label for="THIS" match.
    To prevent confusion, I recommend just using the numbers you gave your paragraphs, like "1.0", "1.1", "2.0" and so on.

    <b>text</b> - makes text bold
